# Ontology Dev Tool — v2 Improvements

These changes build on the existing tool at `/dev/ontology`. No files in `content/` or production routes are touched.

## Part 1 — Graph: visual cardinality on edges

Currently edges show `propertyName (one/many)` as a text label. This is weak — users can't tell at a glance which end is "many".

**Plan:** Replace text-only labels with **crow's foot style SVG arrowhead markers** using React Flow's custom edge type system.

- Create `app/dev/ontology/_components/CardinalityEdge.tsx` — a custom React Flow edge that renders the existing `label` text plus a crow's foot end marker on the target side for `many`, or a single arrow for `one`.
- Register it in `SchemaGraph.tsx` as `edgeTypes = { cardinality: CardinalityEdge }` and set `type: "cardinality"` on each edge built from `RelationshipEdge`.
- The `RelationshipEdge.cardinality` field (`"one" | "many"`) already exists in [`lib/dev/types.ts`](lib/dev/types.ts) — no parser changes needed.

```
one-to-many  (Person → Organization):  ——→ |  (single arrow → crow's foot)
many-to-one  (Meeting → Person[]):     >>——→   (crow's foot → single arrow)
```

The SVG is drawn inline in the edge component using React Flow's `getStraightPath` / `getBezierPath` and a custom `<marker>` definition via `<defs>` in the SVG.

## Part 2 — Rename + scope fix: "Type inspector" → "Schema inspector"

**Changes:**

- Rename tab label: `"Type inspector"` → `"Schema inspector"` in [`OntologyTool.tsx`](app/dev/ontology/_components/OntologyTool.tsx).
- Rename file and component: `TypeInspector.tsx` → `SchemaInspector.tsx`, component `TypeInspector` → `SchemaInspector`.
- **Scope fix**: the inspector currently shows doc sections (a third group not visible in the graph). Remove doc sections from the inspector list. The schema graph only renders entity and event types — the inspector should mirror this. Doc sections appear only in the Coverage tab.
- Rename `TypeDetailPanel.tsx` → `SchemaDetailPanel.tsx` to match.

## Part 3 — Content coverage: clearer labels and explanations

The current table has a "Files" column with a bare number and a "Status" badge. The meaning of each status is not explained anywhere.

**Changes in [`ContentCoverage.tsx`](app/dev/ontology/_components/ContentCoverage.tsx):**

- Rename column header "Files" → "MDX files"
- Add a legend above the table explaining each status:
  - **Populated** — the doc path exists in `content/` and has `.mdx` files (generated by `update-docs`)
  - **Empty** — the folder exists but has no `.mdx` files (pipeline ran but produced no output for this type)
  - **Missing** — no folder exists yet; run `update-docs` to generate it
- Add a short instruction line: "Run `update-docs` to generate content for missing or empty paths."
- Remove doc sections from coverage (or keep them but label them "section" clearly in context).

## Part 4 — New tab: "Source distribution"

Shows how `src/sources/` maps onto the ontology — useful once ingest has run.

**New file:** `app/dev/ontology/_components/SourceDistribution.tsx`

**Data loading:** Add `getSourceDistribution(repoRoot)` to [`lib/dev/parse-ontology.ts`](lib/dev/parse-ontology.ts):

- Walk `src/sources/documents/`, `src/sources/chats/`, `src/sources/transcripts/`
- Parse YAML frontmatter from each `.md` file using the already-installed `yaml` package
- Extract: `source_type`, `people` (array), `orgs` (array), `tags` (array)
- Return:
  - Counts by source type (document / chat / transcript)
  - Entity mention counts — how many sources reference each entity type by matching `people` / `orgs` fields to known entity type names
  - Total sources

**Add a new type to [`lib/dev/types.ts`](lib/dev/types.ts):**

```ts
export type SourceDistribution = {
  total: number;
  bySourceType: Record<string, number>;
  entityMentions: Record<string, number>; // entity type name → count of sources mentioning it
};
```

**UI:** Two small sections:

1. Summary cards: total sources, breakdown by type (document/chat/transcript)
2. Table: entity type → how many sources reference it in `people`/`orgs` fields

When `src/sources/` is empty (fresh template), show a friendly "No sources ingested yet — run the `ingest` skill to populate `src/sources/`" callout.

## Part 5 — New tab: "About this ontology"

Gives users context on what the starter ontology is, why it's structured this way, and how to extend or replace it.

**Two-layer approach:**

**Layer 1 — per-type descriptions from YAML.** Extend the schema YAML format (and parser) to support an optional `description` field on each entity/event type:

```yaml
entity_types:
  - name: Person
    description: "A human individual. The foundation of most knowledge graphs — people participate in events and belong to organizations."
    properties: …
```

Update `lib/dev/parse-ontology.ts` to read `description` from each type and store it in `EntityType`/`EventType`. Display it in `SchemaDetailPanel` and in the About tab per-type listing.

**Layer 2 — `src/ontology/about.md`.** A new Markdown file (created as part of this plan) that serves as the human-readable narrative for the ontology. If it exists, the About tab renders it as HTML. If it doesn't exist, the tab shows a default placeholder explaining what to write there.

**New file:** `app/dev/ontology/_components/AboutOntology.tsx` — renders the about.md content (parsed server-side via `fs.readFileSync` + a lightweight markdown renderer, or as pre-rendered HTML passed from the server component).

**Tab structure for the About view:**

- Top section: rendered `about.md` content (or placeholder)
- Bottom section: a compact table of all entity/event types with their `description` field

## Part 6 — Update the starter ontology in `ontology.mdc`

The current starter (Person, Organization, Meeting) is a good base but:

- Has no `description` fields
- Only has `Meeting` as an interaction type — misses the broader "interactions are the primary data feed" insight

**Changes to `.cursor/rules/ontology.mdc` YAML:**

1. Add `description` to Person, Organization, Meeting
2. Rename `Meeting` → keep `Meeting` but also add `Interaction` as a broader event type with properties: `type` (enum: meeting, email, call, purchase, invoice, receipt), `date`, `participants`, `notes` — so users see the pattern for extending this
3. Or alternatively: keep Meeting exactly as-is but add an `Interaction` event type alongside it with a clear `description` saying "base type for any interaction between people and organizations — extend with specific interaction types"

The cleaner approach is adding `Interaction` as a sibling event type with `type` as an enum hint, leaving `Meeting` as the concrete example. Users can delete either.

**Also create `src/ontology/about.md`** — the starter documentation text explaining:

- What Person + Organization + Interaction represents (a CRM-like foundation)
- That interactions (meetings, emails, calls, purchases) are the primary incoming data feed
- That users can extend, customize, or delete these types entirely
- Links to the `create-ontology` skill for extending the schema

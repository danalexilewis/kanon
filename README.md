# Kanon

## Knowledge system template

A Cursor-driven template for building a **knowledge system** (not a code project): immutable ingest, canonical sources, a user-defined ontology (schema), and **processed output in `content/`** served as a Fumadocs site.

**Data flow:** ingest → sources → docs (shaped by the ontology).

**Skill run order:** create-ontology → ingest → update-docs. Define the schema before you ingest — your ontology decides what the knowledge base _is_; ingest fills it.

---

**Kanon** is Swedish for the same word as English _canon_ — and it has three meanings that fit this project:

1. **A canon of work** — The body of accepted, authoritative sources. This template is built around canonical sources in `src/sources/` and a clear ontology.
2. **A cannon** — In Swedish, _kanon_ is also the word for the big guns on pirate ships. We launch our content into the world like cannonballs :)
3. **Great** — Colloquially, _kanon_ means “great” or “really good” (e.g. _Det var kanon!_ = “That was great!”).

**How to say it (Swedish accent):** Stress the first syllable: **KAH-nohn**. The _a_ is an open “ah” (like in “father”), and the _o_ is a long Swedish _o_, somewhat like the vowel in “go” but more rounded.

## Content development workflow

**For non-technical users and anyone building the knowledge base.** You add and edit material under **`src/`**. The **ingest** skill may also download images and audio to **`public/media/`** (you do not edit those by hand). **`content/`** is generated output — preview it in the dev site, but do not hand-edit it to stay in sync.

### Where things live

| Path              | Purpose                                                                                                            |
| ----------------- | ------------------------------------------------------------------------------------------------------------------ |
| `src/ingest/`     | Drop raw material (transcripts, notes, dumps). **Append-only** — add new files; do not edit existing ones.         |
| `src/sources/`    | Canonical Markdown sources. Edit directly or let **ingest** populate them.                                         |
| `src/references/` | Reference material (PDFs, docs). Read-only; used by **create-ontology**.                                           |
| `public/media/`   | Media library (written by **ingest**). Each item in `YYYY-MM-DD_slug/` with `meta.json`. Video URLs stay external. |
| `src/ontology/`   | Optional glossary and README. The _schema_ lives in `.cursor/rules/ontology.mdc`.                                  |
| `content/`        | **Derived** Fumadocs output — generated by **update-docs**, not hand-edited for sync.                              |

### Skill run order

Run skills in this order. This is not the same as data flow — raw material still moves ingest → sources → docs, but you define the ontology _before_ ingesting so sources land in the right shape.

1. **Create ontology** — Add references to `src/references/`, then run **create-ontology** to define the schema in `.cursor/rules/ontology.mdc`. Use a high-reasoning model.

   ```text
   /create-ontology [your instructions]
   ```

2. **Ingest** — Add raw files to `src/ingest/`, then run **ingest** to normalize them into `src/sources/` (and download media to `public/media/`).

   ```text
   /ingest
   ```

3. **Update docs** — Run **update-docs** to generate `content/` from sources and ontology.

   ```text
   /update-docs
   ```

4. **Preview** — `pnpm install && pnpm run dev` → [http://localhost:3000/docs](http://localhost:3000/docs)

### Source of truth

If `content/` disagrees with `src/sources/` or `.cursor/rules/ontology.mdc`, **`src/sources/` and the ontology rule win**. Run **update-docs** to regenerate — do not patch `content/` by hand.

### Ingest protection

`src/ingest/` is append-only. If an agent edits non-manifest files there, a hook **reverts** those edits and writes to `.cursor/next-step.md`. Only `manifest.json` and `manifest.md` are updated by the ingest skill.

### Restore playbook

1. Canonical truth: `src/sources/`, `.cursor/rules/ontology.mdc`, `src/ingest/manifest.json`
2. Regenerate: run **update-docs**
3. Do not manually edit `src/ingest/**` (except manifests) or `content/**` when it disagrees with sources

### Cursor reminders

After relevant file edits, hooks write pipeline reminders to `.cursor/next-step.md` (e.g. run **ingest**, **update-docs**). Check that file when an agent session starts.

## Developer workflow

**For people extending the template, Fumadocs app, or agent configuration.**

### Quick start

```bash
pnpm install
pnpm run dev    # Turbopack — http://localhost:3000/docs
pnpm run build  # Webpack (Serwist compat)
```

### Stack

- **Next.js** 16, **React** 19
- **Fumadocs** 16 (fumadocs-ui, fumadocs-core); fumadocs-mdx 14
- **Tailwind** v4; **Serwist** for PWA/offline
- **TypeScript** throughout

Key paths: `app/docs/` (Fumadocs UI), `lib/source.ts` (content loader), `source.config.ts` (MDX collection: `content/`).

### `.cursor/` architecture

Designed for **token efficiency** — rules load only when relevant; memory is automatic.

#### Rules (`.cursor/rules/`)

No always-apply rules. Two activation modes:

| Mode                | When it loads                         | Examples                                                           |
| ------------------- | ------------------------------------- | ------------------------------------------------------------------ |
| **Glob-scoped**     | Matching files enter context          | `sources.mdc` → `src/sources/**`, `update-docs.mdc` → `content/**` |
| **Agent-requested** | Agent detects intent from description | `plans.mdc`, `corrections.mdc`, `sources-corrections.mdc`          |

`folder-contract.mdc` defines the directory contract for `src/`, `content/`, and `public/media/`.

#### Skills (`.cursor/skills/`)

Three-tier loading: metadata (description) → instructions (`SKILL.md`) → references (`references/` on demand).

| Skill             | Trigger                                 |
| ----------------- | --------------------------------------- |
| `ingest`          | New files in `src/ingest/`              |
| `create-ontology` | Define/refine schema                    |
| `update-docs`     | Regenerate `content/`                   |
| `plan`            | Create/update plans in `.cursor/plans/` |
| `commit`          | Group changes into clean commits        |

#### Hooks (`.cursor/hooks/`)

| Hook                         | Event           | Purpose                                 |
| ---------------------------- | --------------- | --------------------------------------- |
| `after-file-edit.js`         | `afterFileEdit` | Pipeline reminders → `next-step.md`     |
| `revert-ingest-if-edited.js` | `afterFileEdit` | Revert non-manifest `src/ingest/` edits |

#### Memory (`AGENTS.md` + continual-learning plugin)

Durable preferences and workspace facts are maintained automatically in root **`AGENTS.md`** by the **continual-learning** plugin (enabled in `.cursor/settings.json`). No manual memory file or memory-reminder hook.

Content corrections: `.cursor/corrections.md` (for `content/`) and `src/sources/corrections.md` (for ingest normalization).

### Git hooks (Husky)

Pre-commit formats staged `.md`/`.mdx` with Prettier and runs `pnpm run validate-links`. See [scripts/README.md](scripts/README.md).

### Build notes

- **`pnpm run dev`** — Turbopack (Next 16 default)
- **`pnpm run build`** — Webpack (`next build --webpack`) because Serwist fails on Turbopack production builds
- Exclude `app/sw.ts` from `tsconfig.json` — its webworker lib breaks `window` in client components

### Repo layout (developer-facing)

| Path               | Purpose                             |
| ------------------ | ----------------------------------- |
| `app/`             | Next.js + Fumadocs app              |
| `content/`         | Generated docs (Fumadocs source)    |
| `lib/`             | Source loader, utilities            |
| `source.config.ts` | Fumadocs MDX config                 |
| `.cursor/`         | Rules, skills, hooks, agents, plans |
| `AGENTS.md`        | Agent instructions + learned memory |
| `scripts/`         | Validation and utility scripts      |
